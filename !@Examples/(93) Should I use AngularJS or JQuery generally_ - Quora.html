      Angularjs Vs Jquery is a bad comparison. Its like comparing a Sports Car with an Off Roader (I do not mean that angular is sports car or that jquery is off roader).

      Both Angular and Jquery addresses problems of web development. But the approach taken by both is different. One of the reasons for this is, Jquery is much older and thus had a very different set of problems to address.

      What Jquery addresses is how to manipulate dom. I find Jquery solutions very look and feel oriented. Jquery enables me to control how my site looks much more than how it behaves.

      Now the problems that angularjs addresses was partly created by Jquery (No fault of jquery). As dom manipulation became programable more and more people wanted it. And serious web portals now wanted to look good and snappy. And this is what created the problem.

      During the pre-jquery days and when jquery was new the size of javascript was relatively small. But with jquery this changed. Obviously improvement in js engines and browser also played a hand in this. Codes written without a proper framework was difficult to maintain.

      Angular addresses this problem. It has very specific portions for each type of code eg. services, controllers, directives etc. though it markets itself as a solution for making SPA but even for non spa site the code is much more maintainable and scalable with angularjs.

      In my view if you are building any thing more than trivial app for browser and if speed is not of high priority (eg. games), I would highly recommend AngularJS (or another Framework).

      jQuery is not an MVC framework, and no, it's not bad practice to use jQuery, it just depends on what you use it for, the size and structure of your specific application, etc.

      It depends on your project, jQuery is a library which makes writing JavaScript eaiser, e.g.,
      JavaScript: 
      document.getElementById("someDiv")
      = jQuery: 
      $('#someDiv')
      ,
      or
      JavaScript: 
      1
2

      var myDiv= document.getElementById("myDIV");
                             myDiv.style.display = 'none';

      =
      jQuery : 
      $('myDiv').hide();



      but Angular is a bit different, Angular once loaded gets only the data from the server and creats/change parts of your page based on that data using JavaScript for that process, secondly the applications created by Angular are called SPAs (Single Page Applications), e.g., Gmail etc.

      You're afraid that if Angular 2.0 is released, your skills as an AngularJs (1.2 or 1.3) pro will be lost, but its not the reality, if you learn AngularJs 1.x correctly much of the API will remain the same and there will be only minor changes in the API (if any), so if only the release of version 2.0 is your only problem, than don't worry and try this lovely Framework.
      CodeCademy now offer great tutorial on AngularJs which will get you up and running with AngularJs.

      BTW a lot of projects don't yet need Angular and jQuery is perfectly enough for those project, if your project is one of those projects than stick with jQuery (but even if you're not using Angular for your upcoming project, learning it isn't a bad thing).

      jQuery library is primarily used for DOM manipulation. Angularjs, on the other hand, tries to take away DOM manipulation out of the picture in your development process - Angular is all about declarative programming. Ideally, you should be doing most of the front-end work using the ng-* directives like ng-view, ng-repeat, etc.

      That doesn't strictly forbid you from doing some DOM manipulation using jQuery. But as a good Angular practice, you should try and create a directive whenever you find yourself doing a lot of DOM manipulation.

      In fact, the whole point and purpose of Google was that the current HTML syntax lacks the required expressiveness which results in messy code in the form of jQuery code. Angularjs is nothing but Google's answer to this lacking feature in HTML. Angularjs tries to compensate HTML for its lack of expressiveness, by introducing the various ng-* tags and keeping the MVC separation between the angular views, controllers and services.

      My personal opinion is to avoid AngularJS like the plague. Because it mixes markup with programming logic, it becomes difficult for non programmers, to work with.  In fact, anybody but a frontend engineer might have difficulty with it.

      When I work, I prefer to let the design team own the markup and the CSS, because frankly, there are people much better at it than I. And it allows them much greater freedom to design as they want. And if later changes in the design call for changes in the markup, they have control of their own destiny.

      It is for smilar reasons I would avoid react as well.

      And as others have said, JQuery is not an MVC framework. But you should learn it because so much code relies on it. It is pretty much the de facto method of accessing the DOM.

      I won't recommend any other frameworks, because I've yet to see one that meets my strict requirements.
